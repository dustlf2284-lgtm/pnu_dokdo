# 범용 Swimlane (Process Flow) 작성 스타일 가이드

이 문서는 웹/앱 프로토타입, 코드, API 명세 등을 분석하여 도출한 프로세스를 Mermaid.js 기반의 스윔레인 플로우차트로 변환할 때 AI 에이전트가 반드시 준수해야 하는 범용 기준입니다. 대상 시스템의 도메인이나 복잡도에 상관없이 일관되고 오류 없는 시각화 결과를 보장합니다.

## 1. 동적 주체(Actor) 추출 및 스윔레인(subgraph) 정의
고정된 주체(예: User, Front, Back)를 가정하지 마십시오. 제공된 데이터를 분석하여 해당 프로세스에 참여하는 실제 주체들을 동적으로 추출하여 스윔레인(`subgraph`)으로 구성합니다.

* 주체 추출 기준:
  * 사용자 역할 (예: 비회원, 관리자, 파트너사)
  * 시스템/모듈 (예: 클라이언트 앱, 인증 서버, 결제 모듈, 외부 API, 하드웨어 기기)
* Mermaid 문법: `subgraph ID [화면에 표시될 주체명]` 형태로 작성합니다.

## 2. 노드 ID 명명 규칙 (충돌 및 문법 에러 방지 - 매우 중요)
Mermaid.js 파싱 에러를 원천 차단하기 위해, 모든 노드의 ID는 반드시 영문 알파벳과 숫자, 언더바(_) 조합으로만 명명합니다.

* 명명 규칙: `[주체 식별자]_[순번]`
  * 예시: 주체가 'ClientApp'이면 `C_1`, `C_2` / 주체가 'PaymentGateway'면 `PG_1`, `PG_2`
* 절대 금지 사항: 노드 ID 자체에 한글, 공백, 괄호, 특수문자를 절대 사용하지 마십시오. (표시되는 텍스트와 노드 ID를 엄격히 분리할 것)
  * ❌ `사용자1[버튼 클릭]`
  * ✅ `U_1[버튼 클릭]`

## 3. 노드 형태 (BPMN 기호 표준)
동작의 성격에 따라 시각적 형태를 다르게 부여하여 가독성을 높입니다.

* 시작 및 종료 (`([ ])` 또는 `(( ))`): 프로세스의 명확한 시작점과 끝점
  * 예시: `U_1([앱 실행])`, `S_9([프로세스 종료])`
* 일반 액션 / 프로세스 (`[ ]`): 버튼 클릭, 화면 렌더링, 데이터 처리, API 요청 등
  * 예시: `U_2[결제 버튼 클릭]`, `S_3[데이터 검증]`
* 조건 / 분기 (`{ }`): 로직이 나뉘는 분기점 (If/Else, Switch)
  * 예시: `S_4{결제 성공 여부}`
* 저장소 / 데이터베이스 (`[( )]`): DB 저장, 파일 읽기/쓰기 등 영구적 데이터 상태 변화
  * 예시: `DB_1[(결제 내역 저장)]`

## 4. 흐름(Arrow) 및 라벨링 규칙
* 순차 흐름 (`-->`): 액션 간의 일반적인 이동
* 조건 분기 흐름 (`-- 텍스트 -->`): 마름모(`{ }`) 노드에서 뻗어나가는 화살표에는 반드시 진행 조건(Yes/No, 성공/실패 등)을 텍스트로 명시합니다.
  * 예시: `S_4 -- 성공 --> DB_1`
* 비동기 / 선택적 흐름 (`-.->`): 백그라운드 작업이거나 메인 흐름을 블로킹하지 않는 서브 작업 (예: 이메일 발송, 푸시 알림)

## 5. 논리적 흐름 연결 원칙
* 단절된 노드가 없어야 합니다. 모든 노드는 시작점에서 출발하여 최소 하나 이상의 종료점으로 연결되어야 합니다.
* UI 인터랙션(버튼 클릭) ➔ 클라이언트 처리 ➔ 서버/API 호출 ➔ 결과 반환 ➔ UI 업데이트의 사이클이 끊어지지 않도록 작성합니다.

## 6. 자체 검증 체크리스트 (Hard Rules)
- [ ] `mermaidCode`가 `flowchart TD` (위에서 아래로) 또는 `flowchart LR` (왼쪽에서 오른쪽으로)로 시작하는가?
- [ ] 동적으로 추출된 주체명들이 `subgraph`로 적절히 그룹화되었는가?
- [ ] 모든 노드 ID(식별자)가 한글, 공백, 특수문자 없이 영문+숫자+언더바 형식으로 작성되었는가?
- [ ] 분기 노드(`{ }`)에서 나가는 화살표에 조건 라벨이 누락 없이 작성되었는가?
